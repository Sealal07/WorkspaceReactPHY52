# Техническое Задание: Разработка B2B SaaS-платформы «Workspace Flow»

Workspace Flow — это интерактивная веб-платформа для управления корпоративным шерингом и бронированием ресурсов в гибридном офисе (рабочие места, переговорные комнаты, парковочные места, техника).

## 1. Технологический стек

- **Библиотека:** React (версия 18+)
- **Стилизация:** Tailwind CSS (основная стилизация через утилитарные классы) + CSS Modules (для изоляции специфичных компонентов, где это необходимо по заданию) + Inline Styles (для динамических вычислений).
- **Маршрутизация:** React Router v6
- **Управление состоянием:** useState, useReducer, useContext
- **Дополнительно:** Обязательная типизация (TypeScript или PropTypes), useRef, useMemo, кастомные хуки, ErrorBoundary.

## 2. Структура страниц и Маршрутизация (React Router)

Приложение должно содержать следующие маршруты:

- **/** (Главная страница / Dashboard): Сводная информация для сотрудника. Ближайшие бронирования, баланс доступных дней/часов, быстрые кнопки.
- **/booking** (Страница бронирования): Интерактивная панель с фильтрами и списком/сеткой доступных ресурсов.
- **/resource/:id** (Детальная страница ресурса): Подробная информация о переговорной или рабочем месте, расписание занятости на день, форма бронирования.
- **/profile** (Личный кабинет): Данные сотрудника, история его бронирований с возможностью отмены.

## 3. Технические требования и реализация React-концепций

Внедрить в проект следующие архитектурные решения.

### 3.1. Стилизация (Комбинация Tailwind, Modules и Inline)

- **Tailwind CSS:** Используется для верстки 90% интерфейса (гриды, флексы, карточки, шрифты, адаптивность).
- **CSS Modules:** Написать минимум 2–3 компонента с использованием модулей (например, кастомный Calendar или сложный Sidebar) для демонстрации навыка изоляции стилей.
- **Inline Styles:** Обязательно использовать для динамических интерфейсных элементов:
  - Процент заполненности офиса (progress bar).
  - Позиционирование и длина занятых слотов времени на таймлайне (вычисление `style={{ left: '${start}%', width: '${duration}%' }}`).

### 3.2. Обработка событий и Условный рендеринг

- **onClick:** Выбор слота времени, отмена бронирования, переключение табов.
- **onChange:** Фильтрация списка ресурсов, поиск по названию, ввод данных в форме бронирования.
- **onMouseOver / onMouseLeave:** При наведении на забронированный слот времени должно всплывать кастомное модальное окно (Tooltip) с деталями (кто забронировал, из какого отдела).
- **Условный рендеринг:** Отображение разных статусов («Подтверждено», «Ожидает», «Отменено»), показ скелетонов (лоадеров) во время симуляции загрузки данных, вывод предупреждений о превышении лимитов.

### 3.3. Управление состоянием (useReducer, useState, useContext)

- **useState:** Локальные триггеры (открытие модалок, текущий поисковый запрос, выбранная дата).
- **useReducer:** Управление «Сессией бронирования» (Корзиной). Сотрудник может выбрать несколько ресурсов перед подтверждением (например: Hot Desk на понедельник + Переговорка на 14:00 + Проектор). Редьюсер должен обрабатывать экшены: `ADD_SLOT`, `REMOVE_SLOT`, `CLEAR_SESSION`, `UPDATE_COMMENT`.
- **useContext:** Создать UserContext для хранения данных о текущем пользователе: его имя, роль (Employee / Admin), отдел и лимиты (например, доступно 20 часов бронирования в месяц).
- **Props-drilling:** Продемонстрировать контролируемую передачу пропсов от контейнера /booking вниз к карточке ресурса и кнопкам действий через 2–3 уровня вложенности (передача callbacks для кликов).

### 3.4. Хуки Жизненного цикла и Оптимизация

- **useEffect:**
  - Симуляция запроса к серверу (через setTimeout на 800мс) для получения списка ресурсов при изменении фильтров или даты.
  - Синхронизация истории бронирований с localStorage (чтобы данные сохранялись при перезагрузке страницы).
- **useMemo:** Оптимизация фильтрации. Если в списке 50+ ресурсов, их фильтрация по типу, этажу и наличию оборудования (например, «нужен монитор Type-C») должна быть обернута в useMemo.

### 3.5. Продвинутые механики

- **useRef:**
  - Автоматический фокус на инпуте поиска при переходе на страницу бронирования.
  - Реализация логики «клик вне области» (Click Outside) для закрытия выпадающего списка фильтров.
- **Кастомные хуки:**
  - `useFetch` — для обработки состояний data, loading, error.
  - `useLocalStorage` — для работы с сохраненными бронированиями.
- **ErrorBoundary:** Обернуть страницу /booking или интерактивную сетку часов. Написать компонент-заглушку, который отобразится, если в данные бронирования переданы некорректные даты (симулировать краш для проверки).

## 4. Бизнес-логика и сценарии использования (Use Cases)

### Сценарий «Поиск и Фильтрация»

Пользователь заходит на /booking, выбирает дату «Завтра», тип «Переговорная» и ставит галочку «Есть флипчарт». Список должен мгновенно отфильтроваться без перезагрузки страницы.

### Сценарий «Пакетное бронирование»

Пользователь кликает на несколько свободных слотов времени в детальной карточке ресурса, они добавляются в боковую панель «Черновик бронирования». Пользователь нажимает «Подтвердить», и баланс его доступных часов уменьшается, а слоты становятся занятыми.

### Сценарий «Защита от дурака»

Система не должна позволять забронировать ресурс на прошедшее время или на время, которое уже перекрыто другим бронированием (валидация на фронтенде внутри редьюсера).
